# meta-add-mcp

## Purpose

Add an MCP server to the OS. MCPs expose structured tool surfaces — external services like GitHub, internal subsystems like the vault — to Claude Code. Two modes:

- **`custom`** (default) — Scaffold a new OS-built MCP at `mcps/<id>/` with manifest, server, deps, and `.env` template. Used when no vendor MCP exists, when we want a tight tool surface, or when the MCP composes external calls with OS-side behavior (event logging, vault writes).
- **`hosted`** — Register a third-party hosted MCP endpoint (vendor-run, OAuth-authed) directly in `.mcp.json`. No code, no folder, no deps — just a config row. Used when a vendor offers a maintained MCP that fits your need (the official GitHub MCP at `api.githubcopilot.com/mcp/` is the canonical example).

See `vault/wiki/_seed/meta/reference/standard-mcp-architecture.md` for the contract and `vault/wiki/_seed/meta/reference/concept-mcp.md` for the plain-language overview.

## When to use

- A skill needs structured access to a service the OS hasn't wrapped yet → start here
- An internal OS subsystem (vault, scheduler, events.db) should be queryable as typed tools → `kind: custom`
- A vendor has a maintained MCP that covers your need (and you trust the surface) → `kind: hosted`

## When NOT to use

- One-off CLI commands → just use Bash
- Anything that renders UI → that's an app (see [[meta-add-app]])
- A markdown procedure for Claude to follow → that's a skill (see [[meta-add-skill]])

## Procedure

Dispatch on `kind`. If `kind: custom` (the default), follow **Section A**. If `kind: hosted`, follow **Section B**.

### Common validation (both modes)

1. `mcp_id` matches `^[a-z][a-z0-9-]*$`.
2. `mcp_id` is not already a key in `.mcp.json`'s `mcpServers`.
3. If `kind: custom`: `mcps/{{mcp_id}}/` does not already exist; `domain` is provided and exists as `domains/<name>/`.
4. If `kind: hosted`: `url` is provided and looks like a valid HTTPS URL.

### Section A — `kind: custom` (OS-built MCP)

1. **Create the directory** `mcps/{{mcp_id}}/`.

2. **Write `manifest.json`** at `mcps/{{mcp_id}}/manifest.json`:

   ```json
   {
     "id": "{{mcp_id}}",
     "domain": "{{domain}}",
     "description": "{{description}}",
     "transport": "stdio",
     "command": "node",
     "args": ["mcps/{{mcp_id}}/server.mjs"],
     "env": [{{ each env_var as quoted, comma-joined }}],
     "tools": [
       {{ for each tool_name in tool_names }}
       { "name": "{{tool_name}}", "summary": "TODO: describe this tool" },
       {{ /for }}
       {{ if tool_names is empty }}
       { "name": "ping", "summary": "Health check — returns { ok: true, ts }." }
       {{ /if }}
     ]
   }
   ```

3. **Write `package.json`** at `mcps/{{mcp_id}}/package.json`:

   ```json
   {
     "name": "agentic-os-mcp-{{mcp_id}}",
     "version": "0.1.0",
     "private": true,
     "type": "module",
     "description": "MCP server: {{description}}",
     "scripts": { "start": "node server.mjs" },
     "dependencies": {
       "@modelcontextprotocol/sdk": "^1.0.0"
     }
   }
   ```

   Add service-specific deps based on the MCP's purpose (heuristic from `description`):
   - GitHub-related → also add `"@octokit/rest": "^21.0.0"`
   - SQL/SQLite-related → no extra dep; use built-in `node:sqlite`
   - Vault-related → no extra dep; read filesystem directly

4. **Write `.env.example`** with one commented row per declared env var. If `env_vars` is empty, write a minimal `.env.example` noting no auth is required.

5. **Write `server.mjs`** following the pattern in `standard-mcp-architecture.md` § 3:
   - Minimal dotenv parser reading `mcps/{{mcp_id}}/.env`.
   - Validate required env; `process.exit(1)` with a helpful message if missing.
   - `TOOLS` array with the requested starter tools (or a `ping` placeholder).
   - `HANDLERS` map with stub implementations returning `{ ok: false, error: 'not implemented' }`.
   - Standard `Server` + `ListToolsRequestSchema` + `CallToolRequestSchema` wiring.
   - `await server.connect(new StdioServerTransport())` at the bottom.

6. **Write `README.md`** with sections: Setup (`npm install` + `.env`), Tools table, Auth model, Local smoke test, Adding a new tool.

7. **Install deps**: `cd mcps/{{mcp_id}} && npm install`. Wait for completion.

8. **Regenerate `.mcp.json`**: `node scripts/sync-mcp-config.mjs`. Confirm the new MCP appears in the output.

9. **Smoke test**: boot the server briefly to catch syntax errors (the server will hang on stdio — that's expected, kill it after a second):

   ```sh
   (GITHUB_TOKEN=test node mcps/{{mcp_id}}/server.mjs < /dev/stdin &) ; sleep 1.5 ; jobs -p | xargs -r kill
   ```

10. **Report**:
    - Path to scaffolded MCP, tools registered, env vars (which still unset).
    - Reminder: author must (a) fill `mcps/{{mcp_id}}/.env`, (b) implement tool handlers (currently stubs), (c) restart Claude Code.

### Section B — `kind: hosted` (third-party endpoint)

1. **Read existing `.mcp.json`** (or initialize `{ "mcpServers": {} }` if missing).

2. **Add a row** to `mcpServers` keyed by `{{mcp_id}}`:

   ```json
   {
     "type": "{{transport || 'http'}}",
     "url": "{{url}}"
   }
   ```

   Most hosted MCPs use `"type": "http"`. Use `"sse"` only if the vendor's docs say so.

3. **Write the merged config back** to `.mcp.json` (pretty-printed, trailing newline).

4. **Run `node scripts/sync-mcp-config.mjs`** to confirm the entry survives the round-trip and that the audit's third-party heuristic (no `command`, has `url`+`type`) keeps it out of stale-warnings.

5. **Run `node scripts/audit.mjs --json | jq -r '.findings[] | select(.id | startswith("mcp-"))'`** — expect zero MCP findings.

6. **Report**:
   - The `.mcp.json` row that was added.
   - That OAuth (if the vendor uses it) happens via Claude Code's `/mcp` command on next restart — the author runs `/mcp` once and completes the browser flow.
   - That if auth fails or the vendor's MCP is unreachable, the author should check the vendor's docs (Claude Code will surface the connection error in `/mcp`'s output).

## Inputs schema notes

- `env_vars`: pass an array of UPPER_SNAKE_CASE names. Each becomes a row in `.env.example` (custom mode). Usually empty for hosted (vendor handles auth via OAuth through Claude Code's `/mcp`).
- `tool_names`: pass an array. Each becomes a `TOOLS` entry + stub handler returning `{ ok: false, error: 'not implemented' }`. Ignored for hosted (the vendor controls the tool surface).
- `url`: only meaningful for `kind: hosted`. Reject the run if `kind: hosted` and `url` is missing.

## Outputs

**Custom mode**: a scaffolded MCP under `mcps/{{mcp_id}}/` that boots, advertises tools to Claude Code, and returns stub responses until the author implements handlers.

**Hosted mode**: a new row in `.mcp.json` pointing at the vendor's endpoint. No local files, no deps.

**Both modes**: `.mcp.json` updated; the author must restart Claude Code for the new MCP to be picked up.
